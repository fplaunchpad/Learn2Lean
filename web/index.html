Verified Pen-and-Paper Arithmetic

The addition and subtraction taught in school, written as programs and proven correct in Lean 4 · source

Formal verification is often seen as a specialised pursuit, the preserve of researchers and seasoned developers. Yet its central concern, that an implementation actually meets its specification, can be raised very naturally through the first algorithms anyone learns: adding and subtracting numbers column by column.

This page takes those two algorithms as case studies. For each one it describes the method, lets you run it a step at a time, and presents a Lean 4 proof that the method is correct for every possible input, not only for the examples you happen to try. It is written so that a reader with no background in Lean can still follow close to every line.

Where code appears, hovering over a highlighted name, symbol, or tactic shows a short explanation. Inside a proof it also shows the goal that remains to be established at that point. This is a small offline stand-in for the interactive view that Lean provides while one works.

① Representing a number

Before anything can be proved about an algorithm, the computer must be told what a number is. We store a number as the list of its digits, least significant first, so 1239 is held as [9, 3, 2, 1]: units, then tens, then hundreds, then thousands. Each digit is a value between 0 and 9, called a Digit, and a list of them is a MultiDigit.

A single function connects this representation to ordinary numbers. toNat reads a digit list and returns the number it denotes.


  
The second line says that the value of a list is its units digit, plus 10 times the value of everything above
     it. This function is the thread running through the whole project, because every theorem is stated in terms of
     toNat. That is how one expresses, precisely, the claim that the digit list an algorithm produces
     denotes the right number.

  
The least-significant-first order is chosen because it matches the order in which the algorithms actually work,
     the units column first, as one does by hand. This keeps both the recursion and the proofs straightforward.

  
Two conveniences recur, and they account for the general style of the proofs that follow. First, a digit is a
     Fin 10, so it is below 10 by construction; there is no separate validity check to carry around,
     and the few table cases that would need a digit of 10 or more are impossible and dismissed at once. Second, the
     tactic omega settles routine arithmetic on its own, including bounds and carry conditions. Relying
     on it is deliberate: it leaves each proof free to address the mathematical structure of the algorithm, the
     induction and the place-value identity, rather than routine calculation.


  
  
② Vertical addition


  
The idea

  
The two numbers are lined up by their right edge and added column by column, beginning with the units. Whenever
     a column total reaches 10 or more, its last digit is written down and a 1 is carried into the column to the
     left. This carry is the only information that passes between columns, and it is exactly what the algorithm and
     the proof must keep track of.


  
The algorithm

  
Addition is built in three layers, and each layer is proved correct before the next is placed on top of it.

  
    
    
  1. The single-digit sums. addTable is a lookup table covering every pair of digits from 0 to 9,
        returning the result digit together with whether the sum carried.
    2. 
    
  2. One column. addDigits adds two digits and an incoming carry, using at most two table lookups:
        it adds the digits, then adds one more if a carry arrived. It receives a carry and also returns one, and
        that symmetry is what allows the single-column adders to be chained into the full algorithm, each column
        feeding the next.
    3. 
    
  3. Whole numbers. verticalAdd walks the two digit lists together, passing the carry out of each
        column into the next.
    4. 
  

  

  
The 0 and 1 above are digits; the short proofs that they lie below 10
     are left out here for readability. When one list runs out, the algorithm continues with an implicit
     0, which is how it adds numbers of different lengths and how a leftover carry can become a new
     leading digit, as in 99 + 1 = 100.

  
Lean also requires a guarantee that the recursion stops. We supply a measure, the combined length of the two
     lists, which decreases at every recursive call, and termination follows. The same measure is used for
     subtraction below.

Why it is correct

  
The proof has the same three layers as the algorithm. addTable_correct checks all 100 table
     entries by exhaustive case analysis, and addDigits_correct checks all 200 single-column cases the
     same way. verticalAdd_correct then proceeds by induction over the digit list: assuming the
     algorithm is already correct on the shorter tail, processing one further column, justified by
     addDigits_correct, keeps it correct. The induction is the column-to-column carry argument, stated
     precisely.

  
The proof brings out one point that is easy to miss. Ordinary addition always begins with no carry, yet the
     theorem is stated for an arbitrary incoming carry. That generality is not for the caller; it is what allows the
     induction to go through, since each recursive step passes the next column a possibly different carry, and the
     inductive hypothesis must hold for every carry value. A short corollary, verticalAdd_correct',
     then specialises the result to the no-carry case, which is the version one uses to add two numbers from the
     start.

③ Vertical subtraction


  
The idea

  
Subtraction has the same column-by-column shape, but in place of a carry it uses a borrow. When the top digit
     in a column is too small, ten is taken from the column to the left before subtracting. There is one condition
     that addition does not have: subtraction is defined only when the top number is at least the bottom one. If it
     were not, the result would be negative, which a list of digits cannot represent. This condition appears in the
     proof as a hypothesis.


  
The algorithm

  
The three layers mirror addition, with a borrow in place of the carry.

  
    
    
  1. subTable gives every single-digit subtraction from 0 to 9, returning the difference digit and
        whether it borrowed.
    2. 
    
  2. subDigits handles one column: two digits and an incoming borrow.
    3. 
    
  3. verticalSub walks both lists, passing the borrow inward. If the top number runs out while the
        bottom still has digits, the precondition has been broken and it returns the empty list.
    4. 
  

  

  
The second case, in which the top number is exhausted while the bottom still has digits, can
     only occur when the top number is smaller than the bottom one, which breaks the precondition. The algorithm
     returns [] as a marker for “undefined”, and the hypothesis b ≤ a in the theorem
     ensures this case is never the genuine answer. Termination follows, as for addition, from the combined list
     length decreasing at each call.

Why it is correct

  
The proof follows the recursive structure of the algorithm closely. The single-column facts are settled case by
     case, and verticalSub_correct is then proved by induction. The point that distinguishes it from
     addition is that the precondition b ≤ a has to be re-established for each recursive call, which is
     the purpose of the small hpre sub-proofs. This is what ensures the borrow passed inward never asks
     a column to subtract more than it holds. The line-by-line commentary in the proof follows.
